# Proposal: `ContextualFactory` — Runtime-Bounded Dynamic Tools

## Problem

`DynamicToolRegistry::register_type::<T>(prefix)` is purely compile-time. The
schema for `T` is generated by `schemars::schema_for!(T)` — a static macro with
no access to instance data or external context. There is no way today to issue a
tool whose JSON schema reflects a runtime value (e.g., `maximum: player_bankroll`).

## Motivation: Betting Amount in strictly_games

When a player enters the betting phase, the server knows their exact bankroll
(e.g. $450). The ideal agent interface:

- Only valid presets are exposed as tools (`preset_50`, `preset_100`, `preset_200`
  but **not** `preset_500` — it exceeds the bankroll)
- The schema on any amount field carries `"maximum": 450`
- The agent structurally cannot overdraw — no validation error, no retry

This is the same "invalid states structurally impossible" philosophy as the
typestate system, applied at the MCP tool boundary.

## Proposed API Addition

### New Trait: `ContextualFactory`

```rust
/// A tool factory that requires runtime context to instantiate tools.
///
/// Unlike [`AnyToolFactory`] (which is object-safe and inventory-discovered),
/// `ContextualFactory` is monomorphized at the call site and registered
/// explicitly. Object safety is not required.
pub trait ContextualFactory {
    /// Runtime context needed to build tools (e.g., `BetConstraints`).
    /// Use `()` for factories that need no context.
    type Context;

    fn instantiate(
        &self,
        prefix: &str,
        context: &Self::Context,
    ) -> Result<Vec<DynamicToolDescriptor>, ErrorData>;
}
```

### New Registry Method

```rust
impl DynamicToolRegistry {
    /// Register a contextual factory with runtime data.
    ///
    /// Unlike `register_type`, this does not use inventory discovery.
    /// The factory and context are provided explicitly at registration time,
    /// typically at the start of a game phase.
    pub fn register_contextual<F>(
        self,
        prefix: impl Into<String>,
        factory: F,
        context: F::Context,
    ) -> Self
    where
        F: ContextualFactory + Send + Sync + 'static,
        F::Context: Send + Sync + 'static;
}
```

## Design Notes

### Why not `&dyn Any` + downcast?

The previous sketch used `fn context_data(&self) -> Option<&dyn Any>` on
`AnyToolSlot`, requiring `downcast_ref` in the factory. This is fragile:
mismatched types panic at runtime rather than failing at compile time. The
associated type approach catches mismatches at the call site.

### Why not extend `AnyToolFactory`?

`AnyToolFactory` is object-safe so it can be stored as `&'static dyn
AnyToolFactory` in `inventory`. Adding an associated type or non-`dyn`-safe
method would break that. The two-track design keeps existing inventory-based
factories untouched.

### `Context = ()` is the no-op case

Any factory that needs no runtime context implements `ContextualFactory` with
`type Context = ()`. The compiler optimizes away the unit reference. This means
`ContextualFactory` is a strict superset of the existing factory concept and
could eventually unify the two tracks if desired.

### Object safety is not needed here

Contextual factories are registered explicitly (not via inventory discovery), so
the registry can hold them as concrete monomorphized entries rather than trait
objects. No `dyn`, no erasure, no downcast.

## Usage in strictly_games

```rust
/// Runtime constraints for bet amount elicitation.
pub struct BetConstraints {
    pub min: u64,
    pub max: u64,         // = player bankroll
    pub presets: &'static [u64],
}

pub struct BetAmountFactory;

impl ContextualFactory for BetAmountFactory {
    type Context = BetConstraints;

    fn instantiate(
        &self,
        prefix: &str,
        ctx: &BetConstraints,
    ) -> Result<Vec<DynamicToolDescriptor>, ErrorData> {
        let mut tools = vec![];

        // Schema-direct fast path: place_bet(amount: u64) with runtime maximum
        tools.push(DynamicToolDescriptor {
            name: format!("{prefix}__place"),
            schema: json!({
                "type": "object",
                "properties": {
                    "amount": {
                        "type": "integer",
                        "minimum": ctx.min,
                        "maximum": ctx.max
                    }
                },
                "required": ["amount"]
            }),
            // handler validates and returns BetAmount
        });

        // Preset tools — only expose valid ones
        for &preset in ctx.presets.iter().filter(|&&p| p <= ctx.max) {
            tools.push(DynamicToolDescriptor {
                name: format!("{prefix}__preset_{preset}"),
                schema: json!({ "type": "object", "properties": {} }),
                // handler returns BetAmount(preset)
            });
        }

        Ok(tools)
    }
}

// At betting phase start:
let registry = registry.register_contextual(
    "bet",
    BetAmountFactory,
    BetConstraints { min: 1, max: player.bankroll(), presets: &[50, 100, 200, 500] },
);
```

## Relationship to `#[derive(Rand)]`

`#[derive(Rand)]` on `BetAmount` is still valuable — it gives a seeded random
generator for simulation, testing, and shuffle/roll features. The static bound
in `#[rand(bounded(1, MAX_BET))]` reflects the game's absolute maximum, not the
per-session bankroll. The two mechanisms are complementary:

| Mechanism | Bounds | Use case |
|---|---|---|
| `#[derive(Rand)]` | compile-time constant | testing, simulation, shuffle |
| `ContextualFactory` | runtime bankroll | live agent betting elicitation |

## Summary

| Property | Value |
|---|---|
| Breaking change? | No — existing API untouched |
| Object safety required? | No |
| Downcast required? | No |
| Inventory discovery? | No (explicit registration) |
| Monomorphized? | Yes — fully typed at call site |
| Unifies with existing path? | `Context = ()` is the degenerate case |
