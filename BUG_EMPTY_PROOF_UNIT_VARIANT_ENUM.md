# Bug: `#[derive(Elicit)]` generates empty proof streams for unit-variant enums

**Discovered by:** strictly_games dogfood repo  
**Affected file:** `crates/elicitation_derive/src/enum_impl.rs`  
**Affected feature:** `proofs`  
**Severity:** High — `validate_proofs_non_empty()` fails at runtime; the entire
`ElicitComplete` proof-composition guarantee is broken for this class of type

---

## Summary

When `#[derive(Elicit)]` is applied to an enum whose variants are all unit
variants (no fields), the generated `kani_proof()`, `verus_proof()`, and
`creusot_proof()` methods return `TokenStream::new()` — an empty stream.
`ElicitComplete::validate_proofs_non_empty()` returns `false` for any such
type, violating the invariant that every `ElicitComplete` type carries a
non-empty proof.

A secondary cascading case compounds the problem: an enum that carries only
field types that are themselves unit-variant enums also produces an empty
stream, because the delegation loop extends by nothing.

---

## Root Cause

In `enum_impl.rs`, the proof methods for a user-defined domain enum are
generated by iterating over all field types across all variants:

```rust
// Collect all field types across all variants for kani_proof
let mut all_field_types: Vec<&syn::Type> = Vec::new();
for variant in variants {
    match &variant.fields {
        VariantFields::Unit => {
            // No fields to verify   ← silently contributes nothing
        }
        VariantFields::Tuple(types) => {
            all_field_types.extend(types.iter());
        }
        // ...
    }
}
```

When `all_field_types` is empty (all-unit-variant enum), the generated proof
body is:

```rust
fn kani_proof() -> elicitation::proc_macro2::TokenStream {
    let mut ts = elicitation::proc_macro2::TokenStream::new();
    // zero iterations — nothing extends ts
    ts   // ← empty!
}
```

There is no fallback. The stream is returned empty.

---

## Affected Cases

### Case 1 — All-unit-variant enums (root cause)

Any enum like:

```rust
#[derive(Elicit)]
enum Player { X, O }

#[derive(Elicit)]
enum Position { TopLeft, TopCenter, TopRight, /* … */ BottomRight }
```

All nine `Position` variants and both `Player` variants are unit variants.
`all_field_types` is empty; proofs are empty.

### Case 2 — Cascading emptiness via delegation

```rust
#[derive(Elicit)]
enum Square {
    Empty,           // unit — contributes nothing
    Occupied(Player) // field type is Player, which is Case 1
}
```

The derive correctly delegates to `<Player as Elicitation>::kani_proof()`.
Since `Player`'s proof is empty (Case 1), the extend adds nothing. `Square`'s
proof stream is also empty despite having a field.

---

## Why This Is a Bug, Not a Design Choice

The existing proof helpers `kani_single_variant_enum` and
`kani_multi_variant_enum` already prove exactly this property for generated
*style* enums — trivial-but-non-empty proofs that assert reachability and
copy-equality of variants:

```rust
// Already in proof_helpers.rs, used for style enums:
pub fn kani_multi_variant_enum(enum_name: &str, default_variant: &str) -> TokenStream {
    quote! {
        #[kani::proof]
        fn verify_player_variants() {
            let s: Player = Default::default();
            let s2 = s;
            assert!(s == s2, "copy preserves equality");
            let _: Player = Player::X;
        }
    }
}
```

The same treatment is appropriate for user-defined domain enums. "No fields"
does not mean "nothing to prove" — it means the proof is about the enumeration
itself (all variants are constructible, copy semantics hold, the type is
inhabited).

---

## Proposed Fix in `enum_impl.rs`

When `all_field_types.is_empty()` after collecting from all variants, fall
back to the variant-reachability helpers rather than emitting an empty loop.

```rust
#[cfg(feature = "proofs")]
let proof_methods = if all_field_types.is_empty() {
    // No field types to delegate to — emit a trivial-but-non-empty proof
    // that witnesses the enum's own variants are reachable and well-formed.
    let name_str = name.to_string();
    let default_variant = variants
        .iter()
        .find(|v| v.is_default)
        .or_else(|| variants.first())
        .map(|v| v.ident.to_string())
        .unwrap_or_else(|| "Default".to_string());
    quote! {
        fn kani_proof() -> elicitation::proc_macro2::TokenStream {
            elicitation::verification::proof_helpers::kani_multi_variant_enum(
                &#name_str, &#default_variant
            )
        }
        fn verus_proof() -> elicitation::proc_macro2::TokenStream {
            elicitation::verification::proof_helpers::verus_multi_variant_enum(&#name_str)
        }
        fn creusot_proof() -> elicitation::proc_macro2::TokenStream {
            elicitation::verification::proof_helpers::creusot_multi_variant_enum(&#name_str)
        }
    }
} else {
    // Existing delegation path
    quote! {
        fn kani_proof() -> elicitation::proc_macro2::TokenStream {
            let mut ts = elicitation::proc_macro2::TokenStream::new();
            #(
                ts.extend(<#all_field_types as elicitation::Elicitation>::kani_proof());
            )*
            ts
        }
        // …verus_proof, creusot_proof same pattern…
    }
};
```

Note: `kani_multi_variant_enum` requires a `Default` variant for the
round-trip assertion. For domain enums that don't derive `Default`, a
simpler helper that only proves constructibility of the first variant is
sufficient — or the existing `kani_single_variant_enum` (renamed with cleaner
semantics) could be extended to accept a list of variant paths.

## Regression Test to Add

After the fix, add to `crates/elicitation/tests/proof_non_empty_test.rs`:

```rust
// Unit-variant enum — previously produced empty proofs (regression guard)
#[derive(Elicit, Debug, Clone, Copy, PartialEq, Default)]
enum TwoState { #[default] A, B }

assert_proofs_non_empty::<TwoState>("TwoState (unit-variant enum regression)");
```

## Verification That the Bug Is Triggered

From `strictly_games`, with the current unpatched derive:

```
$ cargo test --features proofs -p strictly_tictactoe

test tictactoe_types_proofs_non_empty ... FAILED
  Player: proof methods returned an empty TokenStream
test square_proof_contains_player    ... FAILED
test board_proof_contains_square     ... FAILED
```

All three failures trace back to `Player` and `Position` being all-unit-variant
enums. Once the derive emits a non-empty fallback proof for those types,
`Square` (which delegates to `Player`) and `Board` (which delegates to
`Square`) will also become non-empty automatically.
