# Anodized Spec Layer + Type Explorer

**Status:** Planning

## Problem

Elicitation already has three formal verification backends (Kani, Verus, Creusot) and
type-level reflection via `ElicitIntrospect`. The gap: agents can query *structure*
(patterns, fields, options) but not *contracts* (what does it take to construct a valid
`I32Positive`? what invariants does `StringNonEmpty` enforce?).

Dumping full specs inline would overflow most agents' context windows. The fix is a
lazy dictionary: agents pull only the spec slices they need, when they need them.

---

## Architecture

### Two parallel layers

**Anodized (`#[spec]`)** — the enforcement engine.  
Added to constructors in the main `elicitation` crate. Stable Rust, runtime checks,
compiler-validated on every build. For the numeric types, the conditions ARE already
in the Creusot proofs — we just need to write them once as anodized `#[spec]` instead.

**`TypeSpec`** — the agent interface.  
`&'static [SpecCategory]` tree stored entirely as `&'static str` slices. Zero
allocation at query time. Registered globally via the existing `inventory` infrastructure.
For numeric types, generated by a codegen script from Creusot. For complex types, hand-authored.

### The lazy dictionary pattern

```
Agent: describe_type("I32Positive")
→ "A positive 32-bit integer. Spec categories: requires (1), ensures (1)"

Agent: explore_type("I32Positive", "requires")
→ ["value > 0"]

Agent: explore_type("Config", "fields")
→ [("timeout", "I32Positive"), ("retries", "U8NonZero"), ("name", "StringNonEmpty")]
```

No context overflow. Agent builds exactly the spec context it needs.

## Data Structures

```rust
// crates/elicitation/src/type_spec/mod.rs

#[derive(Debug, Clone, PartialEq, Eq, Hash, derive_getters::Getters, derive_setters::Setters, derive_builder::Builder)]
#[setters(prefix = "with_")]
pub struct TypeSpec {
    type_name: String,
    summary: String,
    #[builder(default)]
    categories: Vec<SpecCategory>,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, derive_getters::Getters, derive_setters::Setters, derive_builder::Builder)]
#[setters(prefix = "with_")]
pub struct SpecCategory {
    /// "requires", "ensures", "maintains", "bounds", "fields"
    name: String,
    #[builder(default)]
    entries: Vec<SpecEntry>,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, derive_getters::Getters, derive_setters::Setters, derive_builder::Builder)]
#[setters(prefix = "with_")]
pub struct SpecEntry {
    label: String,
    description: String,
    #[builder(default)]
    expression: Option<String>,
}

// Inventory key for global registration
#[derive(derive_getters::Getters)]
pub struct TypeSpecInventoryKey {
    type_name: String,
    build: fn() -> TypeSpec,
}
inventory::collect!(TypeSpecInventoryKey);
```

Getters return `&str` (borrowing from the owned `String`). Callers never need to
clone for reads. The struct owns the data; the application works with references to it.

## Key Design Decisions

**Why not extract from the existing formal proofs?**  
Checked all three backends: Creusot uses `#[requires(true)]` for complex types (the
"cloud of assumptions" pattern makes the interesting constraints implicit); Verus defines
its own abstracted type versions with non-Rust DSL operators; Kani is fully imperative.
None gives a clean, complete extraction path across the full type surface. Writing fresh
anodized specs and TypeSpec entries together in one pass is less work than a codegen
pipeline that would still need manual fixes.

**Why `inventory` for registration?**  
Already in use for `ElicitToolDescriptor`. No central registry file to maintain. Each
`impl ElicitSpec` calls `inventory::submit!` in its module. Callers receive `&str`
from getters — the struct owns the `String`, the application works with references.

**Why two separate MCP tools?**  
`describe_type` is the cheap "what's available" call — one line per type, bounded
response size. `explore_type` fetches detail for one category. Keeps responses bounded
regardless of spec complexity.

## Open Questions

1. **Category naming**: Should `fields` be a first-class category (for struct types), or
   should field constraints be folded into `requires`? Leaning toward separate `fields`
   since agents often want structure independently of contracts.

3. **Fuzzy match on type names**: Low priority — if agent types `I32Posiive`, suggest
   `I32Positive`. Could use `strsim` crate for edit-distance suggestions.
